## 빌드 0.2.0 신규 작업

### 1. 속성 키워드 시스템

카드 및 적 효과에 적용되는 키워드 기반 속성 시스템입니다.

- 키워드 종류 및 상호작용 규칙 확정 _(설계 필요 — 선결 결정사항 참조)_
- 기존 카드 효과(DealDamage, GainArmor 등)에 속성 키워드 연결
- UI: 카드 툴팁에 키워드 설명 표시

### 2. 유물 시스템 (기본 + 고급)

- 유물 데이터 구조 설계 (ScriptableObject 기반 RelicData)
- 유물 이벤트 훅 아키텍처: 트리거 조건(턴 시작/카드 사용/피해 등)에 따라 효과 발동
- 기본 유물 2~3종 구현
- 고급 유물: 조건부 효과 / 복수 트리거 유물 1~2종

### 3. 메타 진행

런 사이에 유지되는 진행 상태 저장 구조입니다.

- 저장 방식 확정 _(선결 결정사항 참조)_
- 기본 진행 데이터: 덱 구성, 보유 유물, 현재 맵 위치
- 전투 종료 → 보상 선택 → 맵 이동 흐름 연결

### 4. 보상 시스템

- 전투 승리 시 보상 선택 UI (카드 3장 중 1장 선택 or 유물)
- 선택한 카드/유물이 덱/유물 슬롯에 추가

### 5. 맵 이동 (MapSelectScene 활성화)

- MapSelectScene 노드 클릭 → 해당 전투 씬 로드
- 맵 진행 상태 저장/복원

마일스톤 정의에는 없지만, 현재 코드 상태를 보면 빌드 0.2.0에서 같이 처리하지 않으면 이후 빌드에서 더 어려워지는 항목들입니다.

### 6. 상태이상 확장

현재 Weakness 하나만 구현되어 있습니다. 속성 키워드 시스템을 도입한다면 상태이상도 함께 늘어나야 키워드가 의미를 가집니다. `StatusBase`가 이미 공통 클래스로 설계되어 있어(`Assets/Scripts/Character/Status/StatusBase.cs`) 추가 비용이 낮습니다.

- 제안 후보: **Strength**(공격력 증가), **Poison**(매 턴 피해), **Vulnerable**(피해 받는 양 증가)
- 속성 키워드 설계 시 "이 속성이 어떤 상태이상을 유발하는가"를 먼저 정해야 해서, 상태이상 목록과 키워드 설계는 동시에 진행하는 것이 효율적입니다.

### 7. 카드 풀 확장

속성 키워드를 붙일 카드 자체가 더 있어야 시스템이 테스트됩니다. `CardData` ScriptableObject 구조가 이미 갖춰져 있어 카드 추가 자체는 코드 변경 없이 에셋 생성만으로 가능합니다.

- 속성별 카드 2~3장씩, 총 10장 내외를 목표로 확보하면 빌드 0.2.0 데모에 충분합니다.
- 카드 설명은 이미 CSVHelper/StringTable 파이프라인이 있으므로(`Assets/Scripts/DataTable/StringTableManager.cs`) CSV에 추가하면 됩니다.

### 8. 완전한 게임 루프 (시작 → 종료)

현재는 `BattleScene`에서 곧바로 시작하고, 패배/승리 후 씬 재로드로 끝납니다. 유물·메타 진행 작업과 맞물려 있어 이번 빌드에서 같이 묶는 것이 자연스럽습니다.

- **런 시작 화면**: 새 런 시작 버튼 하나만 있어도 충분합니다. 지금 메인 메뉴가 없어서 직접 씬을 실행해야만 플레이가 시작됩니다.
- **패배 시 런 종료 처리**: 현재 패배 시 씬 재시작만 됩니다. 메타 진행 데이터를 초기화하고 시작 화면으로 돌아가는 흐름이 있어야 "런 기반 게임"의 감각이 생깁니다.

### 9. 적 패턴 다양화

현재 EnemyActionPattern은 존재하지만, 스폰 가능한 적이 코드에 직접 삽입되어 있습니다. 스폰 데이터화(이월 항목 4번)와 함께 속성 키워드를 가진 적을 1~2종 추가하면 키워드 시스템의 실질적인 테스트가 가능합니다.
